# B.3 评测体系与 Badcase 分析

> RL 训练有一个天然陷阱：训练 loss 在下降，reward 在上升，看起来一切正常——但模型实际上可能在"骗取"奖励，真实能力在退化（你在 [附录 A](/appendix_common_pitfalls/intro) 学过 reward hacking）。如果只看训练曲线，你发现不了这个问题。
>
> 所以评测不是训练之后才做的事，而是**贯穿训练全过程的反馈回路**。本节按这个思路组织：先讲怎么选 benchmark 来衡量模型的真实能力，然后讲怎么系统化地分析错误（Badcase 分析——找到模型犯错的模式并针对性修复），最后讲怎么把评测自动化地嵌入训练循环，让模型在退化时自动回滚。

## 为什么评测比训练更难？

- 训练 Loss 下降 ≠ 模型变好（reward hacking 的本质）
- 不同 benchmark 之间可能互相矛盾
- 人类偏好评测主观性强、成本高

## Benchmark 选择指南

| 方向     | 推荐 Benchmark                 | 说明                          |
| -------- | ------------------------------ | ----------------------------- |
| 通用对话 | MT-Bench, AlpacaEval           | LLM-as-Judge 自动评测         |
| 数学推理 | GSM8K, MATH, AIME              | 规则可验证（RLVR 的天然场景） |
| 代码     | HumanEval, MBPP, LiveCodeBench | 单元测试验证                  |
| 安全对齐 | TruthfulQA, ToxiGen            | 检测幻觉和有害输出            |
| 多模态   | MMMU, MathVista                | VLM 专属                      |

## 评测流水线

```
训练 Checkpoint
    ↓
批量推理（vLLM 加速）
    ↓
自动评测（规则 + LLM-Judge）
    ↓
人工抽检（高分歧样本）
    ↓
回归测试（防止新能力覆盖旧能力）
```

## Badcase 分析方法

Badcase 分析是工业界最常用、但教程中最少讲的质量提升手段。核心思路：**找到模型犯错的具体模式，针对性地补充训练数据**。

### 第一步：定义"坏"的标准

- 任务完成率 < 阈值（如代码无法运行、数学答案错误）
- 回答质量低于基线模型（退化检测）
- 安全违规（有害内容、隐私泄露）
- 风格不符合预期（过于冗长、过于简短、语气不当）

### 第二步：系统化采样

- 不要只看随机样本——按错误类型分层采样
- 按难度分组：easy/medium/hard 分别看错误率
- 按任务类型分组：代码/数学/对话/创作分别分析

### 第三步：归因分析

每个 badcase 都需要回答：

- **数据问题**：训练数据中缺少这类样本？还是数据中有噪声？
- **奖励问题**：奖励模型给了高分但实际不好？（RM 失效）
- **算法问题**：策略崩溃？KL 散度过大？过度优化？
- **评测问题**：评测指标本身有偏差？（如 BLEU 不反映真实质量）

### 第四步：数据迭代

| 错误模式         | 补救措施                           |
| ---------------- | ---------------------------------- |
| 某类任务表现差   | 针对性补充该类任务的训练数据       |
| 输出格式不规范   | 增加格式相关的规则奖励或 SFT 数据  |
| 推理链有逻辑错误 | 增加带推理过程标注的 PRM 数据      |
| 安全/幻觉问题    | 补充红队对抗数据和安全约束训练     |
| 旧能力退化       | 在新训练数据中混入旧任务的保留数据 |

## 自动化评测闭环

将评测嵌入训练循环，实现全自动化：

```
训练 Checkpoint → 自动评测 → 与基线对比
    ↓                           ↓
指标提升？                 指标下降？
    ↓                           ↓
继续训练               回滚到上一个 Checkpoint
    ↓                           ↓
N 步评测一次 ← ← ← ← ← 调整超参数/数据
```

关键配置：

- **评测频率**：每 N 步训练后跑一次评测（通常每 100-500 步）
- **自动化回归测试**：用固定测试集检测能力退化
- **告警机制**：评测分数低于基线时自动停止训练并通知

## 关键指标

- **Reward 曲线**：训练中 reward 的趋势（但要警惕 reward hacking）
- **KL 散度**：策略偏移程度（太大 = 学偏了，太小 = 没学到东西）
- **Entropy**：策略的探索程度（过早收敛 = entropy 降太快）
- **胜率 (Win Rate)**：新模型 vs 基线模型的人类偏好胜率
- **Pass@K**：代码/数学任务中，K 次采样内通过的比例

## Benchmark 构建方法论

前面讨论了"怎么选 benchmark"和"怎么做 badcase 分析"。但如果现有的 benchmark 不满足你的需求——比如你的模型面向特定行业（医疗、法律、金融），或者你需要评估一种新能力（如[多模态 Agent](/chapter10_agentic_rl/multimodal-agent) 的跨模态推理）——你就需要**自己构建评测集**。

### 设计原则

构建一个高质量的评测集需要满足三个原则：

**覆盖度（Coverage）。** 评测集应该覆盖目标能力的各个维度。以"代码能力"为例，不能只测 Python，还需要覆盖其他语言；不能只测算法题，还需要覆盖系统设计、Debug、代码审查等场景。一个好的评测集应该有一个明确的**能力矩阵**：

```
                简单    中等    困难
Python 算法     20题    30题    10题
多文件项目      10题    20题    10题
Debug 定位     15题    20题    10题
代码审查        10题    15题    10题
```

**去污染（Decontamination）。** 评测集中的题目不能出现在训练数据中——否则模型只是"记住"了答案，而不是真正掌握了能力。去污染的检查方法：

```python
# 用 n-gram 重叠检测训练集和评测集的泄露
def check_contamination(eval_sample, training_data, threshold=0.8):
    """检查评测样本是否与训练数据泄露"""
    eval_ngrams = set(get_ngrams(eval_sample, n=8))

    for train_sample in training_data:
        train_ngrams = set(get_ngrams(train_sample, n=8))
        overlap = len(eval_ngrams & train_ngrams) / len(eval_ngrams)
        if overlap > threshold:
            return True  # 发现泄露

    return False
```

实际工程中，还可以用**语义相似度**（如用 Embedding 的余弦相似度）来检测更隐晦的泄露——比如同一个题目换了个表述方式。

**可扩展性（Scalability）。** 评测集不应该是一次性的。随着模型能力提升，你需要不断添加新的、更难的题目。一个实用的做法是**版本化管理**——每次评测集更新都发布一个新版本（v1, v2, ...），旧版本保留用于回归对比。

### 数据生产流程

```
1. 定义能力矩阵 → 确定需要评测的维度和难度分布
2. 题目生成
   ├── LLM 生成初稿（高效但需验证）
   ├── 专家编写（高质量但成本高）
   └── 从真实场景提取（如用户反馈中的困难 case）
3. 质量校验
   ├── 去重 + 去污染
   ├── 标准答案验证（确保题目本身没有歧义）
   └── 难度校准（用基线模型试做，调整难度分布）
4. 发布 + 维护
   ├── 版本化管理
   └── 定期更新（防止刷题）
```

### 对抗样本构造

除了常规评测，还需要构造**对抗样本**来测试模型的鲁棒性：

- **边界 case**：空输入、超长输入、混合语言、特殊字符
- **陷阱题**：看似简单但有隐藏条件的问题
- **组合任务**：需要同时调用多种能力的复杂任务
- **干扰项**：包含大量无关信息的任务，测试模型的信息筛选能力

对抗样本不需要太多——通常 50-100 个精心构造的对抗样本就足以暴露模型的系统性弱点。它们的价值在于发现常规评测无法覆盖的 failure mode。

### 评测集维护

评测集不是"建好就完事"的。它需要持续的维护：

| 维护活动     | 频率       | 目的                                   |
| ------------ | ---------- | -------------------------------------- |
| 难度校准     | 每季度     | 确保评测集不会因模型进步而"过时"       |
| 新题补充     | 每月       | 跟上新能力需求（如新的工具、新的模态） |
| 去污染检查   | 每次训练前 | 确保训练数据没有泄露到评测集           |
| 标准答案更新 | 按需       | 修正有歧义或错误的题目                 |

一个健康的评测集应该像一个"活"的系统——随着模型能力的进化而不断更新，始终保持对模型真实能力的区分度。
